# 架构设计文档 — Shader's Principle

## 一、什么是架构

架构回答了三个问题：

1. **代码拆成几块**（模块划分）
2. **块和块之间怎么通信**（依赖关系）
3. **每个块内部长什么样**（类的设计）

好的架构让你三个月后还能看懂自己的代码。坏的架构让你今天写的代码明天就改不动。对于你第一个项目来说，架构的目标不是"设计得完美"，而是 **"有意识地设计"**——知道为什么这么拆，比拆得对不对更重要。

## 二、核心设计原则

### 2.1 高内聚（Cohesion）

> **一个模块只做一件事，而且把这件事做好。**

**好的例子**（本项目的 Framebuffer）：

```
Framebuffer 只做一件事：管理 CPU 端的像素数组
  - 分配像素内存
  - 清屏（全部设为某个颜色）
  - 读写单个像素
```

它不管窗口怎么创建，不管三角形怎么画，只管像素存储。这就是高内聚。

**坏的例子**（上一版的问题）：

```
Data.hpp 做了所有事：
  - 顶点数据结构
  - 纹理采样
  - 描述符池内存分配
  - 管线创建信息
  - 着色器函数指针
  → 这个文件改了任何东西都可能影响全局，内聚性极低
```

**自检方法**：能用一句话说清楚这个类的职责，并且这句话里没有"和"字。

### 2.2 低耦合（Coupling）

> **模块之间通过尽量窄的接口通信，一个模块的内部改动不影响其他模块。**

**好的例子**：

```cpp
// Screen 只需要 Framebuffer 给它像素指针，不需要知道内部存储格式
void Screen::present(const Framebuffer& fb) {
    const float* data = reinterpret_cast<const float*>(fb.pixels().data());
    // 上传到 OpenGL...
}
```

Screen 只知道 Framebuffer 有一个 `pixels()` 方法返回 `vector<Color>` 的引用。如果将来你把 `vector<Color>` 换成 `Color*` 数组，只改 Framebuffer 内部，Screen 不受影响（只要 `pixels()` 的返回类型不变）。

**坏的例子**：

```cpp
// Screen 直接访问 Framebuffer 的内部成员
glTexImage2D(..., fb.m_pixelData.data());  // 假设 fb 内部用 m_pixelData 这个名字
```

如果 Framebuffer 把 `m_pixelData` 改名或换类型，Screen 也得跟着改。这就是紧耦合。

**自检方法**：改一个类的内部实现（private 成员），其他类需要改动吗？需要 == 耦合太紧。

### 2.3 单向依赖

> **依赖关系应该是一条有向无环图（DAG），不能循环依赖。**

本项目的依赖方向是从"基础"到"组合"：

```
Types (最底层，不依赖任何自己写的模块)
  ↑
Framebuffer   Shader   (依赖 Types)
  ↑              ↑
Screen        Pipeline (依赖 Framebuffer / Shader)
  ↑              ↑
main (最顶层，组装一切)
```

`A → B` 表示 A 依赖 B（A include 了 B 的头文件，A 知道 B 的存在）。

**铁律**：底层模块绝不 include 上层模块的头文件。Types.hpp 绝不应该出现 Framebuffer.hpp。

### 2.4 数据的归数据，逻辑的归逻辑

数据类型（struct/数据类）和计算逻辑（函数/算法类）分开：

| 类型 | 例子 | 特征 |
|------|------|------|
| 纯数据 | `Color`, `Vertex`, `Framebuffer` | 主要存数据，方法很简单（get/set/构造） |
| 逻辑 | `Shader`, `Rasterizer`, `Pipeline` | 包含算法，处理/转换数据 |

数据类型的改动频率低（加个字段），逻辑类型的改动频率高（改算法）。分开之后改算法不容易误伤数据结构。

## 三、如何分析问题 & 拆分模块

### 3.1 从数据流入手

软件渲染器的本质是一条数据流水线。画出来就是：

```
[顶点数据]   →  [顶点着色器]   →   [屏幕坐标顶点]  →  [光栅化器]   →   [像素片段]  →  [片段着色器]  →  [帧缓冲]   →   [屏幕]
  Vertex       VertShader       Transformed      Rasterizer      Fragment      FragShader     Framebuffer   Screen
```

**拆分方法**：数据每经过一次"形态转换"，就是一个潜在的模块边界。

- `Vertex → TransformedVertex`：顶点着色器是边界
- `TransformedVertex → Fragment`：光栅化器是边界
- `Fragment → Color`：片段着色器是边界
- `Color[] → 屏幕画面`：Screen 是边界

### 3.2 问自己四个问题

给一个功能分配归属时，问：

1. **这个功能操作什么数据？** → 和数据放在同一个模块（高内聚）
2. **这个功能被谁调用？** → 调用方应该只通过公开接口调用（低耦合）
3. **这个功能变化的原因是什么？** → 如果两个功能会因为不同原因被修改，它们应该在不同模块（单一职责）
4. **能不能独立测试这个功能？** → 如果不能，说明耦合太紧

**实例**：假设你想给光栅化器加"背面剔除"功能。

- 操作的数据：三角形的三个顶点 → 属于 Rasterizer
- 被谁调用：Pipeline 在光栅化阶段调用
- 变化原因：修改剔除算法 ≠ 修改光栅化算法，所以背面剔除可以是 Rasterizer 内部的独立函数
- 独立测试：可以给 Rasterizer 喂三个顶点，检查它是否返回空 → 可以独立测试

→ 结论：放在 Rasterizer 模块里，作为光栅化之前的预处理步骤。

### 3.3 拆分的时机

| 信号 | 行动 |
|------|------|
| 一个文件超过 ~150 行 | 考虑拆 |
| 一个类超过 ~5 个 public 方法 | 这个类可能管了太多事 |
| 一个函数超过 ~30 行 | 拆成子函数 |
| 你给变量取名时犹豫了 | 很可能职责不清晰 |
| 改一个功能要动 3 个以上的文件 | 耦合可能太紧，检查依赖方向 |

## 四、本项目的架构总览

### 4.1 模块依赖图

```
                main.cpp
               /    |    \
              /     |     \
         Screen  Pipeline  Shader
           |       |        |
           |       |        |
       Framebuffer |     Types
           \       |       /
            \      |      /
             Types Types Types
```

依赖规则：
- **Types.hpp**：纯数据类型，不依赖任何项目内的其他头文件。只依赖 C++ 标准库和 GLM。
- **Framebuffer**：依赖 Types。像素存储。
- **Shader.hpp**：依赖 Types。定义着色器函数签名（输入什么、输出什么）。阶段 3-4 才会出现。
- **Screen**：依赖 Types + Framebuffer。把 CPU 像素显示到 GLFW 窗口。
- **Pipeline**：依赖 Types + Framebuffer + Shader。组装整个渲染管线。
- **main.cpp**：依赖所有模块。负责创建对象、启动主循环。

### 4.2 每个模块的角色

| 模块 | 角色 | 为什么它是独立的 |
|------|------|------------------|
| `Types.hpp` | 数据字典 | 所有模块都需要的基础类型。单独文件避免循环依赖 |
| `Framebuffer` | 像素容器 | 只管存像素。不管像素从哪来、到哪去 |
| `Screen` | 显示适配器 | 封装 GLFW/OpenGL 细节。如果将来换显示后端（SDL、直接写图片文件），只改这个模块 |
| `Shader` | 着色器接口 | 定义着色器的输入输出契约。具体着色器（如"红色顶点着色器"）是实现细节 |
| `Pipeline` | 流程编排 | 把顶点→着色器→光栅化→帧缓冲串起来。它是"指挥"，不亲自做任何计算 |
| `main` | 组装入口 | 创建对象、启动循环。只有 main 知道具体的 Screen/Pipeline 实现类 |

### 4.3 关键设计决策

**为什么 Framebuffer 和 Screen 是分开的？**

因为它们的修改原因不同：
- Framebuffer 变化原因：像素存储格式变了（比如从 `vector<Color>` 换成纯数组，或者加深度通道）
- Screen 变化原因：显示方式变了（比如从 GLFW 换成 SDL，或者直接写入 PNG 文件）

如果把它们合成一个类，改显示方式时可能不小心破坏像素存储逻辑。分开之后互不影响。

**为什么 Shader 是独立的模块而不是 Pipeline 的内部类？**

因为着色器是渲染管线中最"多变"的部分——你可能写很多种着色器（纯色、纹理、光照）。如果塞在 Pipeline 里，每加一种着色器都要改 Pipeline。独立出来后，Pipeline 只依赖于"着色器函数签名"（一个函数指针类型），具体实现可以随意替换。

**为什么 Pipeline 不直接写在 main 里？**

阶段 1-2 时，渲染逻辑很短（几行代码），写在 main 里确实没问题。但到了阶段 3-4，管线逻辑会膨胀（顶点遍历、视口变换、光栅化、着色器调用），如果全堆在 main 里，main.cpp 会变成几百行的"上帝函数"。提前规划 Pipeline 模块，是为了给未来的复杂度留一个"家"。

## 五、架构检查清单

每当你要添加一个新功能时，过一遍这个清单：

1. [ ] 这个功能属于哪个现有模块？如果不属于任何现有模块，是不是该新建一个？
2. [ ] 新加的代码会让现有模块的职责变模糊吗？（例：Framebuffer 突然开始做纹理采样 → 不对）
3. [ ] 新模块依赖了谁？被谁依赖？依赖方向是否合理？
4. [ ] 能不能独立测试这个新功能？（不启动整个程序，单独编译运行新模块的单元）
5. [ ] 改这个功能时，最少需要动几个文件？（理想情况：1 个模块内部 + 1 个调用处）

## 六、常见架构错误 & 如何避免

| 错误 | 表现 | 怎么避免 |
|------|------|----------|
| **上帝类** | 一个类有 10+ 个 public 方法，什么都管 | 每个类只对应一个名词（Framebuffer/Screen/Shader）。如果类名叫 `Manager` 或 `Context`，警惕 |
| **循环依赖** | A include B，B include A | 画依赖图，确保箭头只朝一个方向。用 Types.hpp 集中放共享类型来打破循环 |
| **过早抽象** | 还没画出一个三角形就设计了 DescriptorSet 系统 | 先写具体实现，等同一个模式出现 3 次再抽象。不要预判未来需求 |
| **实现细节泄露** | 类的 public 接口暴露了内部数据结构（如返回 `vector` 的裸指针） | public 接口应该用 const 引用或值语义，不暴露内部存储方式 |
| **传话类** | 一个类不干活，只把调用转发给另一个类 | 检查是否可以把中间层去掉，让调用方直接调用最终实现 |

## 七、本项目各阶段的架构演进

| 阶段 | 涉及模块 | 架构变化 |
|------|----------|----------|
| 阶段 1 | Types, Framebuffer, Screen, main | 最简单的结构：数据 + 显示 + 入口 |
| 阶段 2 | 同上 + Pipeline（简化版） | Pipeline 引入，包含硬编码的光栅化逻辑 |
| 阶段 3 | + Shader | 顶点着色器从 Pipeline 中抽离为独立模块 |
| 阶段 4 | Shader 扩展 | 加入片段着色器接口 |
| 阶段 5 | 可能新增 Texture, DepthBuffer 等模块 | 按需扩展，每次只加一个模块 |

每个阶段结束时，架构仍然"完整"——模块数刚好够用，没有多余的预留。

---

## 八、总结

记住三条就够了：

1. **一个类只做一件事**（高内聚）
2. **改一个类不影响其他类**（低耦合）
3. **依赖只朝一个方向**（无循环）

其他的都是这三条的推论。碰到不确定的设计选择，回来对照这三条。
